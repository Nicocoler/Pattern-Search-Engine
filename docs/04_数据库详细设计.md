# PSE 数据库详细设计

Version: 1.1  
Date: 2026-07-18

## 1. 设计目标

数据库需要支撑四类核心数据：

- 行情时序数据：股票日 K、周 K、月 K、复权因子、成交额、换手率。
- 计算派生数据：技术指标、特征向量、事件识别结果。
- 业务配置数据：模板、权重、扫描配置、用户反馈。
- 结果数据：扫描任务、排序结果、解释文本、回测结果。

设计原则：

- 行情原始数据与派生数据分离。
- 指标、特征、事件和算法结果必须带版本号。
- 每次扫描保存配置快照，保证结果可复现。
- 大表按 `symbol + trade_date` 建主键或复合索引。
- JSONB 只用于灵活扩展字段，核心查询字段必须结构化。
- **针对 A 股特殊交易状态，必须提供分区及时序高频读写性能支持。**

## 2. 数据库选型

推荐组合：

- PostgreSQL：业务主库。
- TimescaleDB：K 线、指标、特征等时间序列大表。
- Redis：热点查询、任务状态、短期缓存。

第一阶段可以只用 PostgreSQL；数据量上来以后，再把 `daily_bars`、`technical_indicators`、`feature_vectors` 转成 TimescaleDB hypertable。本设计中同时包含 TimescaleDB 和 PostgreSQL 原生 Range 分区方案，供项目选择。

## 3. 核心实体关系

```text
stocks
  -> daily_bars
  -> technical_indicators
  -> feature_vectors
  -> pattern_events
  -> dirty_factors (除权异步重算任务表)

pattern_templates
  -> search_runs
  -> search_results
  -> user_feedback
```

## 4. 表结构设计

### 4.1 stocks

保存股票基础信息。专门补充 A 股板块分类和 ST 状态，用于支持涨跌幅归一化系数（LNF）计算。

```sql
create table stocks (
  stock_id bigserial primary key,
  symbol varchar(16) not null unique,
  name varchar(64) not null,
  exchange varchar(16) not null,
  industry varchar(128),
  concept_tags text[],
  list_date date,
  delist_date date,
  market_type varchar(16) not null default 'main', -- main(主板), cyb(创业板), kcb(科创板), bjs(北交所)
  st_flag boolean not null default false, -- 标记是否为 ST / *ST 股票
  status varchar(16) not null default 'active', -- active(交易中), suspended(临时停牌), delisted(已退市)
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index idx_stocks_exchange on stocks(exchange);
create index idx_stocks_status on stocks(status);
create index idx_stocks_market_type on stocks(market_type);
```

### 4.2 dirty_factors (除权异步重算任务表)

用于捕捉因除权除息导致历史前复权数据改变的 symbol，以触发特征历史重算。

```sql
create table dirty_factors (
  task_id bigserial primary key,
  symbol varchar(16) not null,
  trigger_date date not null,
  old_adj_factor numeric(18,8),
  new_adj_factor numeric(18,8) not null,
  status varchar(16) not null default 'pending', -- pending(待重算), processing(重算中), success(成功), failed(失败)
  error_message text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index idx_dirty_factors_status on dirty_factors(status);
create index idx_dirty_factors_symbol on dirty_factors(symbol);
```

### 4.3 daily_bars (时序分区大表)

保存日 K 行情。

```sql
create table daily_bars (
  symbol varchar(16) not null,
  trade_date date not null,
  open numeric(18,4) not null,
  high numeric(18,4) not null,
  low numeric(18,4) not null,
  close numeric(18,4) not null,
  pre_close numeric(18,4),
  volume numeric(20,2),
  amount numeric(20,2),
  turnover_rate numeric(10,4),
  adj_factor numeric(18,8),
  source varchar(32) not null default 'AkShare',
  created_at timestamptz not null default now(),
  primary key (symbol, trade_date)
);

create index idx_daily_bars_date on daily_bars(trade_date);
```

### 4.4 technical_indicators (时序分区大表)

保存基础技术指标。

```sql
create table technical_indicators (
  symbol varchar(16) not null,
  trade_date date not null,
  indicator_version varchar(32) not null,
  ma5 numeric(18,4),
  ma10 numeric(18,4),
  ma20 numeric(18,4),
  ma60 numeric(18,4),
  ma120 numeric(18,4),
  boll_mid numeric(18,4),
  boll_upper numeric(18,4),
  boll_lower numeric(18,4),
  boll_width numeric(18,6),
  boll_width_delta numeric(18,6),
  macd numeric(18,6),
  rsi14 numeric(18,6),
  atr14 numeric(18,6),
  volume_ma20 numeric(20,2),
  volume_ratio_20 numeric(18,6),
  created_at timestamptz not null default now(),
  primary key (symbol, trade_date, indicator_version)
);
```

### 4.5 feature_vectors (时序分区大表)

保存单日特征向量。特征数量会持续扩展，因此使用 JSONB。

```sql
create table feature_vectors (
  symbol varchar(16) not null,
  trade_date date not null,
  feature_version varchar(32) not null,
  features jsonb not null,
  quality_flags jsonb not null default '{}'::jsonb,
  created_at timestamptz not null default now(),
  primary key (symbol, trade_date, feature_version)
);

create index idx_feature_vectors_features on feature_vectors using gin(features);
```

### 4.6 pattern_templates

保存形态模板。

```sql
create table pattern_templates (
  template_id uuid primary key,
  name varchar(128) not null,
  template_type varchar(32) not null, -- historical(股票历史), abstract(抽象形态)
  description text,
  source_symbol varchar(16),
  start_date date,
  end_date date,
  window_size integer not null default 60,
  event_schema jsonb not null,
  scoring_weights jsonb not null,
  hard_filters jsonb not null default '{}'::jsonb,
  status varchar(16) not null default 'active',
  created_by varchar(64),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

### 4.7 pattern_events

保存事件识别结果。

```sql
create table pattern_events (
  event_id uuid primary key,
  symbol varchar(16) not null,
  event_type varchar(64) not null,
  start_date date not null,
  end_date date not null,
  confidence numeric(8,6) not null,
  evidence jsonb not null,
  feature_version varchar(32) not null,
  event_version varchar(32) not null,
  created_at timestamptz not null default now()
);

create index idx_pattern_events_symbol_date on pattern_events(symbol, start_date, end_date);
create index idx_pattern_events_type on pattern_events(event_type);
```

### 4.8 search_runs

保存每次扫描任务。

```sql
create table search_runs (
  run_id uuid primary key,
  template_id uuid not null references pattern_templates(template_id),
  run_date date not null,
  market_scope varchar(64) not null,
  status varchar(32) not null, -- pending, running, success, failed
  feature_version varchar(32) not null,
  algorithm_version varchar(32) not null,
  started_at timestamptz not null default now(),
  finished_at timestamptz,
  config_snapshot jsonb not null,
  error_message text
);
```

### 4.9 search_results

保存扫描排序结果。

```sql
create table search_results (
  result_id uuid primary key,
  run_id uuid not null references search_runs(run_id),
  symbol varchar(16) not null,
  window_start date not null,
  window_end date not null,
  total_score numeric(8,4) not null,
  rank_no integer not null,
  score_breakdown jsonb not null,
  matched_events jsonb not null,
  explanation text,
  risk_notes text,
  created_at timestamptz not null default now()
);

create index idx_search_results_run_rank on search_results(run_id, rank_no);
create index idx_search_results_symbol on search_results(symbol);
```

### 4.10 user_feedback

保存人工反馈。

```sql
create table user_feedback (
  feedback_id uuid primary key,
  result_id uuid not null references search_results(result_id),
  user_id varchar(64),
  label varchar(32) not null, -- good_match, bad_match, watchlist, ignore
  confidence integer,
  comment text,
  created_at timestamptz not null default now()
);
```

## 5. 千万级时序大表分区设计

当股票历史数据（5-10年）及每日衍生指标和特征入库后，大表数据量会轻松突破 5000 万行。为了在高频读写及范围查询下保持高性能，系统必须采用分区存储设计。

### 5.1 方案 A：TimescaleDB 超表 (首选推荐)

TimescaleDB 是 PostgreSQL 的时序数据库插件，其 `hypertable`（超表）可实现对时序数据的自动按时间块分区（Chunking），极大地提高了范围检索和滑动窗口提取性能。

在表创建后，执行以下脚本转换为超表：
```sql
-- 将日 K 行情表转为按 trade_date，每 1 年一个分区的超表
SELECT create_hypertable('daily_bars', 'trade_date', chunk_time_interval => INTERVAL '1 year');

-- 将技术指标表转为超表
SELECT create_hypertable('technical_indicators', 'trade_date', chunk_time_interval => INTERVAL '1 year');

-- 将特征向量表转为超表
SELECT create_hypertable('feature_vectors', 'trade_date', chunk_time_interval => INTERVAL '1 year');
```

### 5.2 方案 B：PostgreSQL 原生 Range 表分区 (PG11+ 备选)

若不加载 TimescaleDB 插件，可采用 PostgreSQL 原生的基于 `trade_date` 进行 Range 分区的设计。

* 注意：由于分区表无法由子表直接继承主键复合索引，在分区模式下，主键复合索引必须包含分区键 `trade_date`。
```sql
-- 示例：以 daily_bars 为例，定义 Range 分区主表
create table daily_bars (
  symbol varchar(16) not null,
  trade_date date not null,
  open numeric(18,4) not null,
  high numeric(18,4) not null,
  low numeric(18,4) not null,
  close numeric(18,4) not null,
  pre_close numeric(18,4),
  volume numeric(20,2),
  amount numeric(20,2),
  turnover_rate numeric(10,4),
  adj_factor numeric(18,8),
  source varchar(32) not null default 'AkShare',
  created_at timestamptz not null default now(),
  primary key (symbol, trade_date)
) PARTITION BY RANGE (trade_date);

-- 创建子分区（通常在晚间同步任务中，定时提前动态创建下一年的分区表）
CREATE TABLE daily_bars_y2025 PARTITION OF daily_bars FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
CREATE TABLE daily_bars_y2026 PARTITION OF daily_bars FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');
```

## 6. 数据版本管理

推荐版本字段：
* `indicator_version`：例如 `ind_v1.0.0`。
* `feature_version`：例如 `feat_v1.0.0`。
* `event_version`：例如 `event_v1.0.0`。
* `algorithm_version`：例如 `sim_v1.0.0`。

只要指标公式、特征口径、事件规则、评分模型变化，就必须升级对应版本。旧版本特征不应删除，防止历史任务追溯失败。

## 7. 数据质量规则

日 K 入库前检查：
- `high >= max(open, close, low)`。
- `low <= min(open, close, high)`。
- `volume >= 0`。
- `symbol + trade_date` 不重复。
- 停牌日不强行填充为正常交易日。

特征生成前检查：
- 最近窗口长度满足模板要求（如 60 日）。
- 关键指标非空。
- 复权方式一致。
- 异常跳空（如停牌或除权）有质量标记。

## 8. 生命周期与清理策略

长期保留：
- 股票基础信息、`dirty_factors` 记录表。
- 日 K 原始数据。
- 模板。
- 用户反馈。
- 重要扫描结果。

可重算数据（存储空间不足时可按旧版本清理，清理后可重算）：
- 指标。
- 特征。
- 事件。

可短期保留：
- 中间缓存、临时任务状态、调试日志。

## 9. 验收标准

- 所有核心表、时序超表或 PG 原生 Range 分区表可通过 Liquibase / Alembic 迁移脚本创建。
- 股票表包含 `market_type` 与 `st_flag` 字段，默认适配 AkShare 的市场基本面清洗。
- `dirty_factors` 重算表支持完美的个股历史复写触发，避免产生形态数据断层。
- 每次扫描能追溯到模板、特征版本、算法版本和配置快照。
- 大表查询有明确的 `symbol + trade_date` 时序联合索引支持。
